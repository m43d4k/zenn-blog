AIに技術記事を書かせるとき、真似させるべきは語尾や口癖ではなかった。

必要だったのは、筆者が何を試し、何を観測し、なぜその結論に至ったのかという判断の流れである。そこで、22本の技術記事を分析し、判断の流れを残すためのZenn執筆用Agent Skillを作った。

この記事では、特徴の抽出、文体ルールへの変換、出力比較による改訂までの工程をまとめる。

分析対象は、同一投稿者による技術記事22本である。そのため、このAgent Skillが技術記事の執筆全般に使えるとは限らない。

## 作ったもの

実行時に使うSkill本体は、役割の異なる3つのファイルで構成している。

```text
zenn-writer/
├── SKILL.md
└── references/
    ├── style-guide.md
    └── zenn-markdown.md
```

`SKILL.md`には、Skillを適用する範囲と作業手順、実行時に読むファイルを記述した。文体やZenn記法の詳細はここに置かず、`references/`の2ファイルへ分けている。

Agent Skillの基本構造は、[Codexの公式ドキュメント](https://learn.chatgpt.com/docs/build-skills)に従った。

`references/style-guide.md`は、執筆時に必ず参照する文体ルールである。入力材料の扱い、通常の執筆ルール、条件を満たした場合だけ使うルール、避ける表現を分けている。

[`references/zenn-markdown.md`](https://github.com/m43d4k/agent-skills/blob/main/skills/zenn-writer/references/zenn-markdown.md)は、Zennの記事形式とMarkdown記法を確認するためのリファレンスである。

開発中に保存した分析資料、入力サンプル、比較出力は、公開リポジトリには含めていない。

完成したSkillはGitHubで公開している。

@[card](https://github.com/m43d4k/agent-skills/tree/main/skills/zenn-writer)

インストール後は、依頼の中で`$zenn-writer`を指定して明示的に呼び出せる。

## Codexへの指示は工程ごとに分けた

Codexには、サンプルの収集からSkillの作成までを一度に依頼しなかった。収集、観測、採否の検討、ルール化、出力比較を別の工程にし、それぞれの成果物を確認してから次へ進めた。

一度に依頼すると、記事に頻出する表現がそのままルールへ入り、分析中の観測とSkillへ採用する判断も混ざるからだ。そこで工程を分け、各段階で根拠と判断を確認できるようにした。

以降に示す指示例は、プロジェクトに残した成果物から、目的、制約、出力形式を再構成したものである。実際の依頼を逐語的に再掲したものではない。

各工程でCodexへ渡した入力と、出力した中間ファイルは次のとおりである。

| 工程 | 主な入力 | 主な出力 |
| --- | --- | --- |
| 1. 記事を収集する | 投稿者ページ、収集条件 | `samples/<投稿者>/*.md` |
| 2. 文体と構成を横断分析する | 収集した記事 | `analysis/style-analysis.md` |
| 3. 文体ルール候補を分類する | 横断分析結果 | `analysis/style-candidates.md` |
| 4. 文体ルールの初版を作る | 分類済みの候補 | `references/style-guide.md`の初版、`tests/outputs/slugdock-status-filter.md`、`tests/outputs/unmask-style-v1.md` |
| 5. 話者性を再分析して改訂案を作る | 記事、文体ルールの初版 | `analysis/voice-analysis.md`、`analysis/voice-candidates.md`、`references/style-guide-v2.md` |
| 6. 初版と改訂案を比較する | `tests/style-guide-v2-comparison/inputs/*.md`、初版、改訂案 | `tests/style-guide-v2-comparison/outputs/*.md`、同ディレクトリの`README.md` |
| 7. Skillとしてパッケージ化する | 採用した文体ルール、Zenn公式ドキュメント | `SKILL.md`、`references/style-guide.md`、`references/zenn-markdown.md` |

## 1. Browser Useで気に入った投稿者の記事を収集する

出発点は、技術説明の進め方を気に入っていた投稿者の記事だった。複数の記事を横断すれば、構成や説明方法のうち再利用できる役割を抽出できると考えた。

まず、CodexのBrowser Useによる収集を依頼した。投稿者ページからの記事一覧の確認、対象ページの巡回、本文の抽出、ファイルへの保存までを続けて処理できた。

```text
Browser Useを使って、指定した投稿者が公開している技術記事を収集してください。

- 指定した投稿者ページをブラウザで開く
- 記事一覧を確認し、対象記事のページを順に開く
- 対象にする記事の条件と範囲を守る
- 記事本文を1記事1ファイルのMarkdownとして保存する
- ナビゲーションなど、本文ではない要素は分析対象へ含めない
- 重複する記事を保存しない
- 取得できなかった記事があれば、推測で補完せず一覧に残す

保存先はsamples/とします。
この段階では文体分析や要約を行わないでください。
```

## 2. 文章の特徴をそのままルールにしない

収集して整えた技術記事22本を横断し、構成や表現の特徴と、それらが文章中で果たす役割を確認した。

主な確認項目は次のとおりである。

- 導入、概要、機能説明、検証、使用感、結末の並び
- 外部情報、観測結果、筆者判断を示す表現
- 見出し、段落、箇条書きなどの構造
- 常体と敬体、一人称、疑問形、感情表現の使われ方
- 同じ特徴が見られた記事数と例外

たとえば、疑問形の直後に説明や検証が続くなら、その役割は「次に扱う論点を一つに絞ること」と考えられる。この役割を文体ルールにすれば、元の記事の言い回しをまねずに疑問形を使う場面を判断できる。

各特徴には、説明、役割、確認できた記事数、例外、信頼度を記録した。頻出する特徴を機械的に採用せず、どこまで一般化できるかも同時に残した。

この工程では、Codexへ次のような指示を出した。

```text
samples/にある記事を横断分析してください。

- 記事ごとの要約ではなく、複数記事に共通する文体と構成を調べる
- 単語の出現だけでなく、その表現が文章中で果たす役割を示す
- 特徴ごとに、確認できた記事数、例外、信頼度を記録する
- 固有の言い回しや記事内容を再利用しない
- サンプルから判断できない事項は、判断できないと記録する

この段階では、文体ルールの採否やSKILL.mdの仕様を決めないでください。
分析結果だけをanalysis/style-analysis.mdへ保存してください。
```

最後の2行が工程の境界になる。分析と採用判断を同時に行わせず、まず観測記録だけを作った。

## 3. 候補を4種類に分類する

横断分析の結果は、次の4種類に分類した。

- 採用候補
- 条件付き採用候補
- 不採用候補
- 判断保留

「外部情報、観測結果、筆者判断を区別する」「抽象的な評価を具体的な条件へ結び付ける」といった特徴は、根拠と判断の関係を明確にできるため、採用候補とした。

一方、疑問形、独立したまとめ、メリットとデメリットの一覧などは、記事の目的や構成に応じて使い分ける必要があるため、条件付きの候補にした。

技術内容に関係しない筆者の近況から始める構成や、特徴的な口癖、感嘆符の使い方などは不採用とした。これらをルールにすると、表面的な模倣につながるためである。

また、サンプルだけでは意図を判断できない特徴は保留し、推測でルールへ加えないようにした。

この工程では、前の工程で作成した`analysis/style-analysis.md`だけを入力した。

```text
analysis/style-analysis.mdの観測結果だけを基に、
Zenn技術記事の文体ルールとして使える候補を整理してください。

各項目を次の4種類に分類してください。

- 採用候補
- 条件付き採用候補
- 不採用候補
- 判断保留

各候補には、分類理由、期待する効果、想定される問題、
使用条件、元の表現をそのまま使えるか、
役割だけを抽象化すべきかを記録してください。

この段階では文体ルールを変更しないでください。
```

頻出する特徴をそのまま一般化しないよう、採用理由を書かせ、判断保留も選択肢に加えた。

## 4. 初版では正確さと再利用性を優先した

最初の文体ルールでは、問題、説明、観測、判断の関係を追えることを重視した。専門用語は名称、短い定義、記事内での意味の順に導入する。また、技術や設計の利点を書くときは、対応する制約や適用条件も近くに置く。

同時に、特定の文章へ似せるための表現は避けた。一人称、疑問形、感情語、短文を「人間らしさ」の記号として追加すると、入力材料にない体験や反応まで生成する可能性があるためである。

ここで一度、分析対象とは形式の異なる記事を生成し、レビュー記事固有の構成が持ち込まれていないことを確認した。

この工程では、分析資料を直接使わず、前の工程で分類した候補だけを入力した。

```text
分類済みの候補から、Zenn記事用の文体ガイドを作成してください。

- 採用候補は、特定の記事に依存しない規則へ抽象化する
- 条件付き候補には、適用する条件を明記する
- 不採用候補は、避ける表現や構成として整理する
- 保留項目はルールへ採用せず、別の節に残す
- 固有の語彙、口癖、決まり文句を含めない
- 入力にない体験、観測、感情を文章へ追加させない

すべての記事へ同じ構成を強制する固定テンプレートにはしないでください。
作成した文体ガイドはreferences/style-guide.mdへ保存してください。
```

分類した候補を中間成果物として残したことで、不採用の特徴と保留した論点も後から確認できるようにした。

## 5. 話者性を再分析して改訂案を作る

次に、文体ルールの初版と分析結果を照合した。その結果、表面的な模倣を避けるだけでは、入力材料にある筆者の判断過程まで弱くなる場合があると分かった。

そこで、分析対象の22本の記事を再度確認し、今度は「誰が、何を前提に、何を試し、どの観測から判断したか」を追跡した。この工程では、一人称や口語の多さではなく、筆者の見立て、試行、観測、判断が一人の話者の発言としてつながっているかを話者性の基準にした。

再分析では、筆者の発言を一続きに追えるよう、記事の結論に影響する次の情報を残す必要があると分かった。

- 筆者固有の比較基準と経験範囲
- 使用前の見立てと、試行後に評価が変わった理由
- 期待した結果が出なかった試行
- 未試行、判断不能、仕様上の不明点
- 観測から用途判断へ至る因果関係

再分析の結果を反映し、これらの情報を記事に残す規則を文体ルールへ追加した。また、記事全体の構成とは別に、記事で示した事実や結果と、そこから筆者が考えたことを近くに置く規則も追加した。

調べる項目は次のように指定した。

```text
同じ記事群を、話者性という観点で再分析してください。

ここでいう話者性は、一人称や口語の多さではありません。
次の情報を読者が追跡できるかに注目してください。

- 誰の判断か
- 使用前にどう見立てていたか
- 何を試したか
- 何を観測したか
- どの観測で評価が変わったか
- 失敗や未確認事項が用途判断へどう影響したか

現在の文体ガイドと比較し、残る特徴、失われる特徴、
表面的な模倣になる特徴を分けてください。
この段階では文体ガイドを変更しないでください。
分析結果だけをanalysis/voice-analysis.mdへ保存してください。
```

次に、`analysis/voice-analysis.md`と文体ルールの初版をCodexへ渡し、改訂候補の整理を依頼した。

```text
analysis/voice-analysis.mdと文体ルールの初版を基に、
話者性を保つための改訂候補を整理してください。

各候補には次の項目を記録してください。

- 反映に必要な入力材料
- 過剰に適用した場合のリスク
- 改訂後に確認する失敗状態

この段階では文体ルールを変更しないでください。
結果はanalysis/voice-candidates.mdへ保存してください。
```

候補一覧を基に、文体ルールの改訂案の作成を次のように依頼した。

```text
analysis/voice-candidates.mdと文体ルールの初版を基に、
比較用の改訂案を作成してください。

各候補について、必要な入力材料、過剰適用のリスク、
確認すべき失敗状態を踏まえて反映してください。

現行のreferences/style-guide.mdは変更せず、
改訂案をreferences/style-guide-v2.mdへ保存してください。
```

## 6. 初版と改訂案を同じ入力で比較する

比較では、初版と改訂案へ同じ情報を与えるため、共通入力を用意した。元記事の文章は使わず、使用条件、事前の認識、観測、判断、制約、未確認事項を箇条書きにした。

比較には性質の異なる3つの入力を使った。

1. 試行によって事前の見立てが少し変わった例
2. 肯定的な評価に、継続使用で制約が加わった例
3. 事前の見立てや評価変更が存在しない例

各入力から、初版と改訂案の文体ルールを適用した文章を作り、共通入力との対応を確認した。評価基準は次のとおりである。

- 入力にない体験、感情、予想、発見を追加していないか
- 判断に影響する条件や未確認事項を削除していないか
- 「関連」を「原因」にするような意味の強化がないか
- 存在しない評価変更を作っていないか
- 観測と判断の因果を追えるか

3つの入力すべてで、改訂案は共通入力に含めた情報を維持した。考えが変化した2例では、試した内容、得られた結果、それを受けた考えの変化が初版より明確になった。変化がない1例でも、入力にない予想や発見を追加しなかった。

比較結果を受け、改訂案を現行の`style-guide.md`として採用した。

比較には次の指示を使った。

```text
元記事の文章を使わず、事実と判断を分離した共通入力を作成してください。
共通入力には、使用条件、事前の認識、観測、判断、制約、
未確認事項を含めます。
共通入力はtests/style-guide-v2-comparison/inputs/へ保存してください。

同じ共通入力から、初版と改訂案の文体ガイドを使った文章を
それぞれ生成してください。出力の長さと扱う情報量をそろえてください。
生成結果はtests/style-guide-v2-comparison/outputs/へ保存してください。

比較では、読みやすさだけでなく次を確認してください。

- 入力にない体験や評価変更を追加していないか
- 入力にある失敗や留保を削除していないか
- 技術的な意味を強めていないか
- 観測と判断の因果を追えるか

比較結果はtests/style-guide-v2-comparison/README.mdへ記録し、
採用条件を満たすか判定してください。
この工程では現行ファイルをまだ置き換えないでください。
```

## 7. SKILL.mdには執筆手順だけを置く

`SKILL.md`には、次の作業手順を記述した。

1. 依頼から記事の目的と対象読者を把握する
2. 既存記事や実装を確認する
3. 外部資料の内容、実際に確認した結果、筆者の考えを区別する
4. 構成を決め、文体ルールに従って執筆する
5. 文体ルールとZenn記法を確認し、説明の重複や根拠のない断定を除く

`SKILL.md`は作業手順に絞り、執筆時には`style-guide.md`で文体、`zenn-markdown.md`でZenn記法を確認する構成にした。

## 指示を設計するときの要点

一連の工程で重視した点は、次の4つである。

### 入力を限定する

「このファイルだけを根拠にする」と指定し、観測記録から候補一覧、候補一覧から文体ガイドという順に入力を絞った。前工程で除外した特徴を後工程へ持ち込まないためである。

### その工程で決めないことを書く

分析時には採否を決めず、候補整理時にはガイドを変更せず、比較時には現行版を置き換えないよう明示した。Codexへ作業内容だけでなく停止位置も伝えることで、人間が判断する区切りを残せる。

### 中間成果物をファイルに残す

分析、候補、比較結果を別ファイルにした。チャット上の回答だけで進めるより、次の工程が参照する入力を固定でき、後から採用理由や保留事項を確認しやすい。

### 成功条件より先に失敗条件を決める

比較では「人間らしくなったか」ではなく、入力にない体験の追加、留保の削除、意味の強化、存在しない評価変更を不合格条件にした。これにより、抽象的な品質ではなく、入力と出力の対応を確認できた。

## 作って分かったこと

文体ルールを作る過程で、「どんな表現を足すか」よりも、「筆者の結論を支える情報をどう残すか」を考える必要があると分かった。

語尾や口癖を取り出しても、なぜその結論に至ったかは残らない。一方、模倣を避けるために筆者自身の考えまで削ると、何を基準に、何を試し、何を見て考えを変えたのかまで消えてしまう。

残すべきなのは、筆者の比較基準、試したこと、その結果、考えが変わった理由、未確認の点である。これらを結論までつなげれば、特定の言い回しをまねなくても、筆者の発言を一続きにたどれる記事になる。

Agent Skillに必要だったのは、表現の型ではなく、入力から何を残し、何を作らないかを定めることだった。
